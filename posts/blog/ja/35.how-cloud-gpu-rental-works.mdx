# クラウドGPUの借り方：ゼロからLLM起動まで

> **全体の流れ：** 正確なGPU、region、image、storageを選び、instanceを起動して保護し、software環境をpinし、smoke testを実行します。出力をephemeral storage以外へ保存し、computeを停止して、予測額と実請求を照合します。

このチュートリアルでは次を扱います。

- GPU、region、image、storage lifecycleの選択。
- 安全な接続と環境情報の記録。
- PinしたLLMの起動とendpoint test。
- Output保存とcheckpoint recoveryの検証。
- 課金resourceの停止とusage照合。

クラウドGPUレンタルは、modelが入るGPUを選び、remote environmentを起動し、接続、実行、保存、停止する流れです。commandよりもstorage、access、billing stateが再現性を左右します。

## ステップ1：GPU選択前にワークロードを定義する

model/revision、precision/quantization、最大context、concurrency/batchを書きます。modelがloadできても、長いpromptや複数requestでKV cacheが不足する場合があります。

## ステップ2：Region、GPU、Image、Storageを選ぶ

Glows.ai consoleで利用可能な構成を選び、checkoutでGPU数とvariantを確認します。preconfigured imageでもCUDA、framework、container versionを記録します。

| データ | 保存先の考え方 |
| --- | --- |
| 消してよいpackage cache | Runtime storage |
| 再利用するmodel weight | 永続storage／再現可能download |
| 学習dataset | 管理された永続storage |
| Checkpoint | 永続storage＋外部backup |
| 最終output | 永続storage／download copy |

## ステップ3：起動して環境を記録する

[instance作成ドキュメント](https://docs.glows.ai/docs/create-new)に従い、launch時刻、region/GPU、image/digest、CPU/RAM/storage、接続可能時刻、課金開始を記録します。

**証拠ゲート：** 公開版には実際のGlows.ai画面と記録が必要です。測定前に固定起動時間を主張しません。

## ステップ4：安全に接続する

SSH private keyをlocalに保ち、未認証のinference portをinternetへ公開しません。

```shell
nvidia-smi
python --version
```

Dockerでは`latest`だけでなくimmutable digestを記録します。

## ステップ5：LLMを起動する

```shell
vllm serve <model-id> \
  --host 0.0.0.0 \
  --port 8000 \
  --max-model-len <tested-context-length>
```

placeholderをGPUに入るmodel/contextへ置換し、productionではmodel revisionとvLLM versionをpinします。

```shell
curl http://127.0.0.1:8000/v1/models
```

local endpointで確認し、代表requestのTTFT、総latency、VRAM、warningを記録します。

## ステップ6：必要な状態を保存する

model、data、config、outputが再現可能か、runtime storageに何があるか、必要なstateをpersistent storageへ移したか確認します。学習はcheckpoint restore、推論はcontainer/model/config保存を行います。

## ステップ7：Compute停止と料金確認

applicationを正常終了し、必要データを保存してからconsoleでstop/deleteします。compute停止後もpersistent storageが残り、別課金になる場合があります。

**証拠ゲート：** 公開版で予想runtimeとGlows.ai usage recordを照合し、account/payment情報を匿名化します。

## ステップ8：Recoveryを試す

環境を再起動／再作成し、保存したmodel、config、outputを復元します。run recordにはmodel revision、engine、container、GPU/region、storage、test、costを残します。

## 再現可能なInstanceを作る

記録されていないterminal historyだけで動くserverは復旧しにくく、再実行も高くつきます。Container digest/base image、Python package、CUDA、driver、framework、model/tokenizer revisionをpinします。Secretはimageに入れずplatform mechanismを使い、startup scriptでstorage mount、GPU check、service startを実行し、launch command、health check、expected outputを一緒に保存します。

最小のreproducibility testは、最初のinstanceを見ず、文書だけで2台目を起動することです。同じsmoke-test requestを処理できなければ再現可能ではありません。

## 高価なジョブの前に復旧訓練を行う

1. 小さなcheckpoint/output artifactを作る。
2. Intended persistent locationへcopyする。
3. 計画どおりdisposable computeをstop/deleteする。
4. Pinした環境からfresh instanceを起動する。
5. 保存dataをattach/restoreする。
6. Jobをresumeしてoutputを比較する。
7. どのcompute/storage meterが動いていたか確認する。

Credential不足、ephemeral path、checkpoint incompatibility、未文書化stepを小さな請求で発見できます。長時間trainingのcheckpoint頻度は、失ってよい有料作業量から決めます。

## 操作手順の情報源

Instance手順は最新の[Glows.ai作成ガイド](https://docs.glows.ai/docs/create-new)で確認します。vLLM commandとflagはpinしたreleaseの[公式vLLMドキュメント](https://docs.vllm.ai/)を参照してください。本文のsmoke test、persistence test、billing reconciliationは運用上の確認であり、すべてのimageやinstance typeの動作を保証しません。

## よくある質問

### 起動に何分かかりますか？

inventory、image、region、構成によるため、選択構成で測定します。

### vLLMではなくOllamaを使えますか？

model/workloadを支援すれば可能です。vLLMは高throughput、Ollamaはlocal workflowの移行に便利です。

### Stop後に何が起きますか？

billingとdata persistenceはprovider state modelによります。残るstorageと料金を確認します。

### Port 8000を公開できますか？

authenticationとnetwork controlなしでは公開しません。local test後にaccess layerを追加します。
