# gpt-oss-120Bは1 GPUで動く？3機種のコスト検証

> **既知の事実：** OpenAIはgpt-oss-120Bが想定formatでsingle 80 GB GPUに収まると説明しています。ただし、安全なmemory headroom、実用concurrency、魅力的なtoken costは未証明です。このprotocolでA100、H100、RTX PRO 6000を検証します。

有用なgpt-oss-120B benchmarkは最高tokens/secのscreenではなく、exact model revisionを動かし、task evaluationとlatency limitを通る最安構成です。

OpenAIはgpt-oss-120bがnative quantizationでsingle 80 GB GPUに収まると説明しています。ただし、production concurrency、p95、各構成のcostは実測が必要です。

## 100万Tokenコスト

`100万output token cost = GPU時間単価 ÷ accepted output token/時 × 1,000,000`

prefill/decodeを分けるためinput tokenも報告し、saturated throughputとreal offered loadを別計算します。

## 結果を過大解釈しない

H100が最大throughputでも、valid throughputがprice premiumを上回る場合だけ経済的です。RTX PRO 6000の96 GBはfitを簡単にしますがlatency勝利を保証しません。A100が低価格でSLOを通れば最安の場合があります。結論はversion、prompt、dateに限定します。

## Benchmarkを再現する

container digest、launch command、model/tokenizer、匿名化prompt-length distribution、load config、raw JSON/CSV、cost worksheet、invalid note、evaluation summaryを公開します。

## 1枚のGPUに収まっても本番結果ではない

Model load成功はweightとstartup allocationが収まることだけを証明します。Production claimにはgraph capture/warmup後のheadroom、advertised input/output boundary、target simultaneous sequences、TTFT/TPOT/p95/error rate、quantization/runtimeのquality、sustained memory、restart/cache rebuild/recoveryが必要です。

Single-request latencyから始め、offered loadを段階的に増やし、error/latency thresholdで停止します。User-facing SLOを満たさないsaturated throughputをinteractive capacityとして引用しません。

## 読者が再計算できる結果を公開する

Input token、warmup inclusion、invalid runも示し、tested traceのfit/latency/cost winnerだけを結論にします。
