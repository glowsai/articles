# H100 vs A100 vs RTX PRO 6000：Qwenに最適なのは？

> **判断基準：** A100は低価格なdatacenter baseline、H100は高throughput候補、RTX PRO 6000は96 GBのsingle-GPU候補です。最適解は、対象Qwen modelが収まり、latency目標を満たす構成の中で総コストが最も低いものです。

H100、A100、RTX PRO 6000は、Qwen推論で異なる制約を解きます。H100は新しいdatacenter throughput、A100は成熟したbaseline、RTX PRO 6000は96 GB VRAMと低い表示料金が特徴です。

正しい勝者は、対象Qwen modelを実行し、latency目標を満たす最小総コスト構成です。

## Qwen構成を先に固定する

model family/size/revision、BF16／FP8／INT8／4-bit、context、input/output token分布、concurrency、arrival rate、TTFT／TPOT／p95、serving engineを記録します。concurrency 1でloadできても、production loadでKV cache不足になる場合があります。

## Memory容量が判断を変える理由

`必要VRAM = 重み + KV cache + runtime workspace + 安全余裕`

RTX PRO 6000の96 GBはquantizationやKV cacheの余裕を作れます。H100が高速でも、memory不足でtensor parallelが必要ならコストと障害面が増えます。

## 同条件Benchmark方法

model/tokenizer、vLLM/SGLang commit、container、precision、context、trace、warmup、run durationを固定します。1、10、50、target in-flight requestを測り、repeat varianceを報告します。

## 有効なLatency SLOでコストを計算する

`100万output tokenコスト = GPU時間単価 ÷ 有効output token/時 × 1,000,000`

有効とはTTFT、TPOT、error、qualityを満たすことです。offered loadとprompt distributionを併記します。

## 各GPUを最初に試す場面

### A100から始める

確認したvariantにmodelが入り、成熟したsoftwareと低い完了コストが期待できるbaselineです。

### H100から始める

厳しいlatency、高concurrency、対応precisionが重要な場合に試します。speedupが価格差を上回る必要があります。

### RTX PRO 6000から始める

96 GBでsingle-GPU構成が簡単になる場合に試し、bandwidthや運用特性も測ります。

## テスト前にQwenワークロードを定義する

| Input | 公開するfield | 結果が変わる理由 |
| --- | --- | --- |
| Model | Repositoryとimmutable revision | Architecture/weightでfitとspeedが変わる |
| Precision | BF16、FP8、AWQなど | Memory、kernel、qualityが変わる |
| Prompt mix | p50/p95 input/output tokens | Prefill-heavyとdecode-heavyを分ける |
| Load | Request rate、concurrency | Batchingとqueueingを決める |
| SLO | TTFT、TPOT、p95、error rate | 使用可能なthroughputを定義する |
| Quality gate | Task set、acceptance threshold | 高速だが劣化した構成を除外する |

Interactive traceとthroughput traceを最低1つずつ測ります。長prompt prefillで勝つGPUがdecodeで負けることや、saturated throughputが高くても実loadのtail latencyを満たさないことがあります。

## 勝者を3種類に分ける

- **Fit winner：**安全なmemory headroomを持つ最も単純な有効構成。
- **Latency winner：**error/quality limitを満たしp95が最低。
- **Cost winner：**指定loadのaccepted million tokens単価が最低。

最後にworkload別で推奨します。Controlled result tableが埋まるまで、H100、A100、RTX PRO 6000の優位性はhypothesisです。

## よくある質問

### H100は必ずA100より速いですか？

新しいGPUですが、model、precision、engine、batch、latency制約で実現差は変わります。

### 96 GBならRTX PRO 6000が最良ですか？

model fitには有利ですが、最小latency／最大throughputを保証しません。

### 大型Qwenを1 GPUで提供できますか？

size、quantization、context、concurrencyによります。

### Input／output tokenのどちらでコストを出しますか？

可能なら分けます。Prefillとdecodeの負荷は異なります。
