# SGLang vs vLLM：Productionで選ぶべき推論エンジン

> **正直な答え：** 永続的で普遍的な勝者はいません。両projectは急速に変化し、結果はmodel architecture、engine revision、prompt mix、concurrency、kernel、hardwareで変わります。同一testでcorrectness、tail latency、recovery、upgrade gateを通過したengineを選びます。

SGLangとvLLMは高速に進化するLLM inference engineです。winnerはmodel、release、kernel、GPU、prompt、structured output、concurrencyで変わります。実際に運用するworkloadでcontrolled testします。

throughputだけでなくAPI compatibility、startup、observability、model coverage、failure recovery、upgrade riskを比較します。

## SGLang vs vLLM概要

| 領域 | 確認内容 |
| --- | --- |
| Model support | exact model/quantization |
| API | 必要route/response behavior |
| Performance | TTFT/TPOT/p95/throughput/error |
| Memory | Peak VRAM/KV cache |

| Operations | Metric/log/startup/recovery |

[SGLang repository](https://github.com/sgl-project/sglang)と[vLLM document](https://docs.vllm.ai/)で同じ日付の機能を確認します。

## Controlled Benchmark Method

model/tokenizer、quantization、container/runtime、GPU/CPU/RAM/region、context/output、prompt distribution、client/network/warmup/durationを固定します。documented engine settingのみ調整し、out-of-box/tunedを分けます。

## Workload-Specific FeatureをTestする

long shared prefix/cache、structured JSON、LoRA、tool call、multimodal、tensor parallel、cancel/streamを追加し、speedだけでなくquality/schemaを検証します。

## Operational Fit

cold/warm startup、readiness、metric、log、config、image size、download、graceful termination、crash recoveryを測ります。両engineでupgrade/rollback rehearsalを行います。

## Cost比較

`100万output token cost = GPU時間単価 ÷ valid output token/時 × 1,000,000`

同じGlows.ai hardware/offered loadで、saturatedだけでなくreal utilizationを計算します。

## 選び方

exact model/featureを支援し、measured targetとoperational riskで勝つ方を選びます。engine-neutral harnessを残し、new releaseを再testします。1 modelの結果で全社winnerを決めません。

## 両推論エンジンに同じテスト基盤を使う

同じrequest body、arrival time、prompt/output distribution、cancellationを送るclientを作り、engine-specific settingはversioned configにします。Model、tokenizer、quantization、GPU、driver、container base、warmup、network path、run durationを固定します。

Output validity/task evaluation、TTFT、TPOT、p50/p95、accepted input/output tok/s、error/timeout/cancellation/OOM、peak VRAM、cache utilization、startup/recovery、realistic/saturated loadのcost per million accepted tokensを収集します。各conditionを複数回走らせvarianceを示し、prompt orderをrandomizeします。

## 速度だけでなく本番適合性を評価する

結果を見る前にownerがweighted scorecardを承認します。Latency-sensitive productはp95/recovery、batch pipelineはthroughputを重くできます。Model coverage、API compatibility、observability、upgrade effort、documentation、team familiarityも含めます。

Hard gateはscoreより優先し、output corruption、required model path不足、安全にrecover不能なら高速でもwinnerではありません。

Benchmark前にpass thresholdを記録します。たとえばcontract suiteでschema regressionがゼロ、p95 TTFTがproduct target未満、error rateが合意した上限未満、recoveryがdeployment budget内という条件です。すべてのhard gateを通過した構成だけでcostを比較し、rejectしたrunと理由もresult packageに残します。これにより、好みのengineが一つのmetricで勝った後にthresholdを動かすことを防ぎ、次のreleaseも同じproduction-ready定義で比較できます。

## 推論エンジンの情報源

Feature、flag、対応hardware、release behaviorは公式[SGLangドキュメント](https://docs.sglang.ai/)と[vLLMドキュメント](https://docs.vllm.ai/)で確認します。2026年の研究[The Silent Hyperparameter](https://arxiv.org/abs/2605.19537)は、model weightが同じでもinference backendがreproducibilityに影響し得ることを示すため、correctnessをthroughputと同等に評価します。
