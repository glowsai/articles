# Can One GPU Run gpt-oss-120B? A 3-GPU Cost Benchmark

> **What is known:** OpenAI says gpt-oss-120B can fit on a single 80 GB GPU in its intended format. That does not prove safe memory headroom, useful concurrency, or attractive cost per token. This protocol tests those production questions across A100, H100, and RTX PRO 6000.

The useful gpt-oss-120B GPU benchmark is not the highest tokens-per-second screenshot. It is the lowest-cost configuration that runs the exact model revision, passes a task evaluation, and stays within a declared latency limit under realistic load.

OpenAI describes gpt-oss-120b as an open-weight model designed to fit on a single 80 GB GPU through its native quantization format. That fit claim does not establish production concurrency, p95 latency, or cost on every 80 GB configuration. Those require measurement.

## Benchmark Questions

The test must answer:

1. Does the exact model configuration load with safe memory headroom?
2. What TTFT and TPOT does one interactive request receive?
3. How does p95 change at 10, 50, and target concurrency?
4. What throughput remains valid under the latency SLO?
5. What does that throughput cost per million tokens?
6. Does output quality remain acceptable under the chosen runtime settings?

## Cost per Million Tokens

Use only throughput that meets the declared service target:

`cost per 1M output tokens = hourly GPU cost ÷ accepted output tokens per hour × 1,000,000`

Also report input tokens because prefill cost can dominate long-document workloads. A blended “tokens per second” number can hide whether the test mostly measured input processing or decoding.

Calculate at two loads:

- **Saturated throughput:** enough queued work to keep the GPU busy.
- **Realistic offered load:** the expected arrival pattern, including idle time.

The second number is usually more useful for a production budget.

## Interpret the Result Without Overclaiming

H100 may produce the highest throughput, but it wins economically only if its valid throughput offsets its hourly premium. RTX PRO 6000 may simplify fit with 96 GB, but memory capacity does not guarantee lower latency. A100 may remain economical if it meets the SLO at a lower rate.

The conclusion must be conditional on model revision, engine version, prompt distribution, and date. Repeat the benchmark after material software changes.

## One-GPU Fit Is Not a Production Result

A successful model load proves only that weights and startup allocations fit. A production claim also needs:

- safe memory headroom after graph capture and warmup;
- the advertised input/output boundary;
- target simultaneous sequences without OOM;
- acceptable TTFT, TPOT, p95 end-to-end latency, and error rate;
- stable output quality under the selected quantization and runtime;
- sustained operation without memory growth;
- restart, cache rebuild, and recovery timing.

Test single-request latency first, then increase offered load in controlled steps. Stop a stage when the error or latency threshold fails. Do not quote throughput from a saturated run as interactive capacity unless the same run passes the user-facing SLO.

## FAQ

### Can gpt-oss-120B run on one GPU?

OpenAI states the model can fit on a single 80 GB GPU in its intended format. Actual serving headroom, context, and concurrency still need measurement.

### Is H100 the cheapest per token?

Not automatically. Divide its price by accepted throughput at the target latency and compare the same workload.

### Why include RTX PRO 6000?

Its 96 GB capacity makes it a relevant single-GPU candidate. The benchmark determines whether performance and cost fit the use case.

### Can I compare these numbers with another website?

Only if model revision, quantization, engine, GPU variant, prompts, concurrency, and latency rules match closely.
