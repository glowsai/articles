# How Much Does It Cost to Run an LLM? The Real Formula

> **Core formula:** LLM unit cost is total compute and storage spend divided by accepted work—requests, users, or tokens—that meets the latency and quality target. Hourly GPU price alone misses utilization, idle time, retries, input/output mix, and rejected performance.

The cost to run an LLM is not just a GPU hourly rate or an API token price. For self-hosted inference, cost depends on model fit, measured throughput at the required latency, traffic shape, idle capacity, storage, and operations. For an API, it depends on input/output tokens, caching, model choice, and any platform charges.

The honest answer is a range tied to a documented workload. This guide provides the formulas needed to calculate it.

## Four Useful Cost Units

| Unit | Best for |
| --- | --- |
| Cost per GPU-hour | Infrastructure budgeting |
| Cost per request | Product unit economics |
| Cost per million tokens | Comparing inference configurations |
| Monthly cost at target traffic | Capacity and architecture decisions |

Do not compare providers using different units until they are normalized to the same workload and quality requirement.

## GPU Runtime Cost

Start with:

`GPU cost = hourly rate × active seconds ÷ 3,600`

Glows.ai listed GPU rates from $0.49/hour for RTX 4090 to $5.00/hour for B200 on July 20, 2026. The selected GPU must fit the model and meet the latency target; the lowest rate is not automatically the lowest cost per output.

Add persistent storage, transfer, restart overhead, and operational labor where they apply.

## Cost per Million Tokens

After measuring valid throughput:

`cost per 1M output tokens = hourly GPU rate ÷ output tokens per hour × 1,000,000`

Suppose a configuration costs $1.20/hour and sustains 60,000 accepted output tokens per hour at the required p95 latency. Its compute component would be $20 per million output tokens. This is a formula example, not a Glows.ai benchmark result.

Report input and output distributions because prefill and decode use hardware differently. Also publish the concurrency and latency threshold that made a result “accepted.”

## Cost per Request

For a stable workload:

`cost per request = total serving cost during period ÷ successful requests`

Include failed and retried requests in the cost numerator. If a server processed 10,000 successful requests but attempted 10,500, the failed capacity still cost money.

For API pricing:

`API request cost = input tokens × input rate + output tokens × output rate + other billable features`

Use the provider’s current prices and do not compare a proprietary model with an open model until an evaluation set shows both meet the same product requirement.

## Utilization Changes Everything

Peak benchmark throughput assumes enough queued work to keep the GPU busy. Production traffic is rarely that smooth.

| Traffic pattern | Cost behavior |
| --- | --- |
| Continuous high demand | Better GPU utilization; capacity may be economical |
| Business-hours demand | Idle nights/weekends raise effective cost |
| Bursty traffic | Warm reserve improves latency but increases idle cost |
| Offline batch | Queue can improve utilization if deadlines allow |

A $1/hour GPU used productively only 25% of the month has an effective productive-compute cost closer to $4 per utilized hour before storage and operations.

## Capacity Planning From Requests

Collect these inputs:

- requests per second and peak arrival rate;
- input/output token percentiles;
- context length;
- TTFT, TPOT, and p95 targets;
- measured tokens per second at multiple concurrency levels;
- replica count and recovery reserve;
- scale-up and scale-down delay.

Then simulate hourly demand rather than multiplying peak throughput by 730 hours.

**Evidence gate:** The publication version must ship with an editable calculator plus Glows.ai results for at least three offered-load levels. Include the exact model, engine, quantization, GPU, prompt distribution, commands, and raw aggregates.

## Costs That Belong in the Model

- model and dataset storage;
- image download and cold-start time;
- observability and gateway components;
- replicas held for recovery;
- engineering and on-call work;
- data transfer;
- failed deployments and retries;
- evaluation required after model or engine upgrades.

The model should have a “known cost,” “estimated cost,” and “not measured” column. False precision is worse than an honest range.

## API or Your Own GPU?

An API often wins for small, unpredictable traffic and when the managed model’s quality is required. A dedicated GPU can win at sustained, predictable utilization when an open model passes the evaluation set. A hybrid design can route baseline traffic to owned capacity and overflow or special tasks elsewhere.

The crossover is not a universal monthly request number. It changes with token lengths, batching, latency, model quality, and idle policy.

## Three Monthly Cost Scenarios

The same GPU can produce radically different unit economics:

### Bursty internal assistant

Suppose the service receives brief weekday peaks and is almost idle overnight. A dedicated GPU billed continuously has low utilization, so cost per accepted request rises even if peak throughput looks excellent. An autoscaling or scheduled design may reduce spend, but cold start and model load must remain within the product’s latency tolerance.

### Steady production endpoint

A stable request stream can keep continuous batching effective. Measure accepted output tokens per hour at the actual prompt distribution, then apply:

`monthly compute = hourly GPU rate × active GPU hours × replica count`

Add the spare capacity required for p95 latency and failure recovery. Running at 100% theoretical utilization is not a safe production budget.

### Offline document pipeline

If documents can wait in a queue, fill larger batches and shut the GPU down after the queue drains. Cost per document is:

`(compute + storage + failed-run cost) ÷ accepted documents`

This workload may favor a faster, more expensive GPU when shorter runtime more than offsets the hourly premium.

## Build a Cost Sheet That Finance Can Audit

For every estimate, preserve:

- price, currency, region, GPU variant, and timestamp;
- model, engine, precision, context, and prompt/output distribution;
- offered load, concurrency, latency SLO, error rate, and accepted throughput;
- active, idle, warmup, failed, and recovery time;
- persistent storage, transfer, and shared-service allocation;
- monthly traffic assumption and peak multiplier.

Present a base, low, and high case. The sensitivity range is more honest than a single cost-per-token number and shows which assumption deserves the next benchmark.

## Cost Method Sources

The formulas follow basic capacity accounting and require measured serving output. The 2026 paper [Beyond Per-Token Pricing](https://arxiv.org/abs/2606.11690) documents why offered load and utilization can dominate self-hosted LLM unit economics. Use the [official vLLM benchmark suite](https://docs.vllm.ai/en/latest/benchmarking/) or an equivalent pinned load generator, then preserve raw inputs, rejected runs, and the exact pricing timestamp.

## FAQ

### How much does an LLM cost per hour?

The compute rate can range widely by GPU, but total hourly cost includes storage and supporting services. Use a GPU that fits the model and measure useful throughput.

### How do I calculate cost per million tokens?

Divide hourly cost by accepted tokens per hour, then multiply by one million. Publish the latency and workload assumptions beside the result.

### Is self-hosting always cheaper than an API?

No. Low utilization, engineering overhead, and quality differences can make an API cheaper. Self-hosting becomes attractive only under a specific workload and operational model.

### Should I count idle GPU time?

Yes. Idle reserve is part of the service cost unless the architecture actually scales down without breaking the latency target.
